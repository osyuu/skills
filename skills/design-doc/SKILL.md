# Requirement → Detailed Design Document

把需求變成一份**輕量契約 + 驗收清單**，給後續的 plan 與 code 當依據，並在實作後當驗收標準回頭檢查。

## 核心理念（先讀，這決定品質）

設計書對 AI 生成 code 的真正價值，**不是文件好看，而是在寫 code 前把模糊點逼到收斂**。AI 生成最大的不穩定來源是需求欠規格（underspecified）：需求一模糊，模型就自己腦補、中途漂移。設計書的工作就是把這些腦補點變成白紙黑字的決定。

所以這份文件的定位是**介面與行為層的契約**，不是實作說明書。要避開三個傳統「詳細設計書」的陷阱：

1. **不要寫虛擬碼 / 逐函式實作**——那等於把 code 抄一遍，馬上過時、又綁死實作，AI 反而被綁住。實作交給 code。
2. **不要把粒度寫到行級**——寫到「這個介面收什麼、吐什麼、錯誤怎麼處理、狀態怎麼轉」就停。
3. **詳細要詳在對的地方**——資料模型、介面契約、**狀態轉換、邊界條件、錯誤處理、驗收標準、非目標**。這些是 AI 最常猜錯、且不會隨實作細節變動而立刻過時的部分。

判斷一行內容該不該寫的準則：**「如果不寫下來，AI（或別的工程師）會不會猜，而且可能猜錯？」** 會 → 寫。不會（顯而易見或純風格） → 省略。

### 五個讓設計書真正有用的原則（業界共識 + SDD）

前四條來自 Google design doc、ADR、Rust/PEP/KEP/React RFC 等公開實務；第五條來自 spec-driven development（spec 驅動 AI 實作）的教訓：

1. **設計書的存在理由是「取捨與被否決的替代方案」**。只描述你選的那個解法，那叫實作手冊，不是設計書。一定要寫「考慮過哪些做法、為什麼選這個、放棄了什麼」——這是文件最有價值的部分，不能省。
2. **逼問「為什麼不要做這個」(drawbacks) 與「不做會怎樣」(do-nothing baseline)**。強迫自我批判：實作/維護成本、複雜度、有沒有現成套件就能解。這能殺掉過度設計。
3. **先寫問題、獨立於解法 (constraint-first)**。背景與目標要能「即使你的設計被推翻，仍然成立」。把約束條件寫清楚，不要綁死在你選的解法上——這樣文件壽命更長。
4. **右尺寸，必要時不寫**。不是每個需求都要整份設計書。解法顯而易見、低風險 → 別寫，直接做或寫幾行 plan 即可。風險高、不確定、跨團隊、工程量大 → 才值得完整寫。長度隨風險縮放：小需求 1~2 個重點章節就好，別硬湊滿模板。**先判斷「這需求值不值得寫設計書」，再決定寫多細。**
5. **（SDD）spec 是給實作者的工單，也要持續是單一真相**。實作 AI 拿這份文件工作時，不該有任何需要「發明語意」的地方；文件要列全**交付物**與**可執行的驗收方式**；實作偏離或決策翻轉時，**同一批改動就回寫文件**（版本 +1），不讓 spec 與 code 漂移。⚠️ 護欄：SDD 補強止於此——**不要**引入 requirement ID、追溯矩陣、EARS 句型等重形式；右尺寸原則（第 4 條）優先於完整性。

## 工作流程

### Step 0 — 取得需求

確認需求來源並讀進來：
- **禪道 (ZenTao) story / bug / task**：若使用者給了 story ID，用對應的 zentao skill 取得內容（如 `zentao-story`）。
- **貼上的文字 / issue / 既有規格檔**：直接讀。
- 來源不明確就先問：「需求在哪？story ID、檔案路徑、還是直接貼給我？」

### Step 1 — 抽取 + 反問（最關鍵的一步，不要跳過）

讀完需求後，**先不要急著生成**。列出需求中的**模糊點、隱含假設、缺漏**，整理成幾個有重點的問題回問使用者。這一步是整個 skill 的價值核心——收斂發生在這裡。

問題要**聚焦在「不問清楚就會猜錯」的地方**，不要問顯而易見或可合理預設的東西（那種直接帶預設值、在文件裡標註即可）。用 `references/clarify-checklist.md` 當提問靈感清單，挑真正有缺口的面向問，通常 3~6 題、一次問完。

提問時：
- 對能合理推定的部分，**直接給預設答案讓使用者確認/否決**（降低使用者負擔），例如「我假設未登入者不能存取，對嗎？」
- 真正開放、影響設計走向的，才當開放題問。

使用者回答後若又冒出新的模糊點，可以再問一輪，但別無止盡——抓到「足以下筆且不會大幅猜錯」就進下一步。剩餘的未知放進文件的「待確認事項」。

**拿不到答案時的 fallback（非互動場景）**：若是排程、批次、headless 等無法即時得到使用者回答的情境，不要卡住。改成：把原本要問的問題**連同你採用的預設答案**寫進文件開頭的「釐清問題與假設」表，關鍵的腦補點就地標 `【假設】`，然後依這些 stated assumptions 把設計書寫完；真正影響設計走向、需要拍板的，再列進「待確認事項」。原則是**讓所有替使用者做的決定都可見且可被推翻**，而不是停下來等。

### Step 2 — 生成設計書

用 `assets/design-doc-template.md` 當骨架產生內容。重點：

- **先右尺寸**：依需求的風險與不確定性決定要寫多細（見上面原則 4）。低風險就只填核心幾章；別為了填滿模板而灌水。
- **按需裁剪章節**：模板涵蓋面廣，但**不是每個需求都要填滿所有章節**。沒有狀態機就不要硬湊狀態轉換；純後端就拿掉 UI 章節。空章節要嘛刪掉、要嘛標 `N/A` 並一句話說明為何不適用——不要留空殼。
- **替代方案不能省**：一定要寫「考慮過哪些做法、為何選這個、放棄什麼」，並寫「不做這件事會怎樣」。這是設計書的核心價值（原則 1、2）。
- **背景要 constraint-first**：把問題與約束寫成獨立於解法也成立（原則 3）。
- **每個被你做的假設與決策都要可見**：寫進「設計決策」或就地標註，讓使用者一眼看到你替他決定了什麼。
- **驗收標準必須可驗證**：寫成「可勾選、可測」的條目（給定…當…則…），這是實作後回環檢查的依據，不要寫成模糊形容詞。
- **圖示慣例**（若畫圖）：關係線要有方向、有標籤（「讀取」「送出訂單」而非含糊的「使用」）、跨服務標協定；每張圖附一句圖例說明。
- **語言**：跟著對話語言（預設繁體中文）；程式碼、識別字、型別、API 名稱、欄位名維持英文。

**SDD 完整性掃描（初稿完成後、寫檔前必跑）**——對照初稿逐項自問，這六個維度是「實作 AI 一定會踩到、需求方卻幾乎不會主動講」的高頻猜測點；有涉及就補進文件，不涉及就跳過（別為掃描而灌水）：

1. **資料生命週期**：每個新 entity 的「父資料被刪/改時怎麼辦」寫了嗎？cascade 清理、孤兒列策略、反正規化 counter / 冗餘欄位的回減，缺一個就是資料 bug。既有的刪除路徑要實際去讀 code 確認現況，不要假設。
2. **自動產生資料的欄位預設**：任何由系統自動建立的 row（backfill、upsert 附帶建立），每個欄位填什麼要白紙黑字（NULL？預設值？誰之後補？）。
3. **限制與 clamp**：分頁上下限、字串長度、數值範圍、格式驗證——沿用專案既有慣例並寫明數字。
4. **跨介面語意一致性**：相似操作（如兩種寫入端點）的驗證/錯誤語意一致嗎？**刻意不同**的地方要寫明「與 X 刻意不同，因為…」，否則實作者會自行類推統一。
5. **顯式接受風險**：刻意不緩解的風險（濫用、無認證寫入、資料膨脹…）逐條寫「接受＋理由＋將來收緊的路徑」，避免 review 時被重新翻案、或實作者自作主張加防護。
6. **自我延續機制的收斂性**：任何會反覆觸發自己的機制（分頁／無限捲動、retry、輪詢、debounce、auto-advance、心跳）要寫明三件事：(a) **每次迭代由什麼驅動**（滑到近尾端？事件？計時器？）；(b) **終止條件與其權威來源**——`hasMore`／停止旗標要來自單一真相（後端 `total`、明確 flag），**不要用「回滿一頁就假設還有」這種需要兩個數字手動同步的啟發式**；(c) **驅動參數取退化值時仍成立嗎**——page size=1／空頁／清單短到無法再滑，觸發還會 fire、終止還會收斂嗎？（踩過：loadMore 只綁滑動觸發，page_size=1 時無下一頁可滑 → 永不觸發 → 卡死。）

同時確認兩件 SDD 交付要求：**交付物清單**列全了嗎（code、migration、**版本化契約文件**、驗證腳本、輔助工具——漏列的交付物 = 不會被做）？**每條驗收標準都標了驗證方式**嗎（自動測試 / 腳本 / 手動步驟；若環境跑不了本地測試，驗證腳本本身要升級為交付物）？

### Step 3 — 寫檔 + git 排除

1. 產出路徑：`docs/design/<slug>.md`（`<slug>` 用 kebab-case 的需求簡稱，例如 `user-login-otp`）。`docs/design/` 不存在就建立。
2. **git 排除（預設，不 block）**：預設把設計書設為**本機個人排除**（當個人設計筆記、不污染團隊共用的 `.gitignore`）。寫入 `.git/info/exclude`（若 repo 是 git）：
   - 檢查 `.git/info/exclude` 是否已含 `docs/design/`，沒有才追加一行 `docs/design/`。
   - 若不在 git repo，提醒使用者並照常寫檔。
   - 這是 least-regret 預設（excluded → 之後要 commit 很容易；反過來收回已 push 的檔是改歷史的痛）。**自動套用、不問使用者**，以維持 headless / 全自動 SDD。
3. 寫完後回報：檔案路徑、關鍵決策、待確認事項，**並明講這個 git 決定 + 一句翻轉指引**——「已加進 `.git/info/exclude`（預設當個人筆記、不進版控）；**若這是團隊共享 spec、或有會 committed 的檔（如 `CLAUDE.md`）要指向它，請從 exclude 移除並 commit**，否則對隊友 / CI 是懸空指標」。

### Step 4 — 銜接 plan（選用）

設計書定稿後，它就是很好的 plan 輸入。若使用者要接著實作，建議進 plan：用設計書的「整體設計 / 資料模型 / 介面契約」推導實作步驟與檔案改動順序。設計書回答 WHAT/結構，plan 回答 HOW/順序，兩者互補。

### Step 5 — 回環驗證 + 回寫（選用但強烈建議）

實作完成後，回頭拿設計書的「驗收標準」逐條檢查實作是否符合。這把設計書變成可驗證的閉環，是「增加開發穩定性」的關鍵——不要只生成、不回收。

**回寫（SDD：spec 保持單一真相）**：實作過程若偏離設計（欄位改名、行為調整、決策翻轉），**在同一批改動裡更新設計書**——版本 +1、文件資訊表加一行變更記錄（一句話講改了什麼＋為什麼）。過期的 spec 比沒有 spec 更危險：它會主動誤導下一個讀它的人（或 AI）。

## 模板

完整章節骨架在 `assets/design-doc-template.md`，生成時讀取它。各章節重點：

| 章節 | 為什麼重要（AI 視角） |
|---|---|
| TL;DR | ≤5 句講清楚要做什麼、怎麼做，讓 reviewer 30 秒進入狀況 |
| 背景與目標 | 防止 AI 做錯方向；記錄「為什麼」；constraint-first |
| 範圍 / 非目標 (Non-goals) | **防止 AI 過度發揮、擴張需求**——最被低估的一節 |
| 整體設計 | 模組分工、資料流；可用 mermaid（關係要有方向+標籤+圖例） |
| 資料模型 | 型別、約束、關聯——AI 最常猜錯 |
| 介面契約 | input/output、錯誤碼、例外行為 |
| 行為與流程 / 狀態轉換 | AI 失誤率最高之處，必須明列 |
| 邊界條件與例外處理 | 否則 AI 只寫 happy path |
| 非功能需求 | 效能/安全/權限/i18n，依需求取捨 |
| **替代方案與取捨** | **考慮過哪些做法、為何選這個、不做會怎樣——設計書的核心價值** |
| 設計決策 (ADR-like) | 脈絡→選項優缺→決定→好壞 consequences→可逆性 |
| 相依與風險 / Drawbacks | 外部相依、已知風險、本方案的缺點與代價 |
| 推出與遷移 (選用) | 破壞性變更、feature flag、灰度、升級/回滾 |
| 交付物 (Deliverables) | **spec 即工單**：code、migration、版本化契約文件、驗證腳本——漏列 = 不會被做 |
| 任務分解 (選用) | 多 agent / 跨 session 執行時，「執行順序」本身是交付物：每條任務指回規格章節、標真依賴（=可並行邊界）與驗證方式；單人單 session 跳過 |
| 驗收標準 | 可測條目 + 各自的驗證方式，實作後的回環檢查依據 |
| 待確認事項 / 未來可能 | 未收斂的未知；out-of-scope 但別丟的點子（parking lot） |

## 提問清單

各面向的提問靈感見 `references/clarify-checklist.md`。用它檢查有沒有漏問的維度，但只問真正有缺口的。

## 品質檢查（生成後自問）

- 這需求**值得**寫這麼長的設計書嗎？有沒有灌水硬湊章節？
- 有沒有寫「考慮過的其他做法 + 為何否決」？有沒有寫「不做會怎樣」？（少了這些，這份就只是實作手冊）
- 有沒有把我替使用者做的假設都標出來？
- 範圍 / 非目標 寫清楚了嗎？AI 照這份做會不會做超過或做不夠？
- 背景是否獨立於解法成立（constraint-first）？
- 驗收標準每一條都「可驗證」嗎，還是混了形容詞？各自標了驗證方式嗎？
- 有沒有寫成虛擬碼 / 抄 code？（有的話刪掉，拉回介面與行為層）
- 不適用的章節是刪掉或標 N/A，而不是留空殼？
- SDD 完整性掃描跑了嗎？（資料生命週期/cascade、自動資料的欄位預設、限制與 clamp、跨介面一致性、顯式接受風險、自我延續機制的收斂性）
- 交付物列全了嗎（含版本化契約文件、驗證腳本）？
- 有沒有為了 SDD 而過度形式化？（requirement ID / 追溯矩陣 / 為掃描而灌水 → 刪）
- 若填了任務分解：每條都指回規格章節而非複述？依賴是真依賴？單人單 session 的需求是否誤填了本節？
- 若設計書含自動化執行計畫（agent 團隊執行）：授權邊界有沒有明列 git 操作粒度（開分支、merge 到預設分支、push、部署）？注意 spec 裡的授權**不等於** harness 權限層的授權——「直推/合併預設分支」需要使用者在對話中明示，寫計畫時預設走 feature branch、合併留給使用者拍板。
