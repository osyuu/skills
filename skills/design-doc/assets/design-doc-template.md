- **外部相依**：第三方服務、套件、其他團隊的介面。

## 13. 設計決策 (Design Decisions)

針對關鍵的、單一的決策，用 ADR 結構記錄，讓未來的人（和 AI）知道「為什麼這樣選」。

### 決策：<標題>
- **狀態**：proposed | accepted | superseded by <ID>
- **脈絡與問題**：要解決什麼、有哪些決策驅動因素（constraints / quality goals）。
- **選項**：考慮過哪些做法。
- **決定**：選了哪個。
- **後果 (Consequences)**：好的（變容易的事）與壞的（變難的事 / 新增的限制）都要寫。
- **可逆性**：這個決定好不好回頭？若是高成本不可逆，特別標註。

填寫準則：
- 每條任務**指回規格章節**（§N），不複述內容——規格只有一份，任務表只管順序與邊界。
- 依賴只寫真依賴：它決定可否並行；無依賴的任務就是 fan-out 單位。
- 每條有自己的驗證方式：完成定義 = 驗證過，不是「寫完了」。
- 顆粒度到「一個 agent 一輪能完成」為止，不再往下拆；不引入 requirement ID / 追溯矩陣。
- **放哪裡**：預設就放本節（單一真相）。當任務多到高頻變動（每天勾選/重排/追加）時，拆到同目錄 `<slug>.tasks.md`，本節只留連結——設計書是版本化契約，任務勾選是 runtime 狀態，後者的變動**不**需要 bump 設計書版本。執行中的即時狀態（誰正在做）不進任何檔案，由 leader session 的任務追蹤管。
