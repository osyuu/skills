# 反問清單（提問靈感）

Step 1 用這份清單檢查「有沒有漏問的維度」。**不是每題都要問**——只挑需求裡真的有缺口、不問清楚就會猜錯的。能合理預設的，直接帶預設值請使用者確認/否決，別當開放題。

## 範圍與目標
- 這次到底要做到哪？哪些明確「不做」？（範圍最容易膨脹）
- 成功長什麼樣？怎麼判定做完了？
- 有沒有時間/版本壓力影響取捨（先求能動 vs 一步到位）？
- **不做這件事會怎樣？**（do-nothing baseline——確認這需求真的值得做，也避免過度設計）
- 有沒有現成做法可參考或直接用？內部既有模組、開源專案、現成套件？（prior art，避免重造輪子）

## 使用者與角色
- 誰會用 / 誰能用？是否分角色、權限不同？
- 未登入 / 未授權的人看到什麼、能做什麼？

## 資料
- 涉及哪些資料 entity？新增還是改既有？
- 關鍵欄位的型別、必填、預設、約束、唯一性？
- 與既有資料的關聯？需要 migration 嗎？對既有資料的影響？
- **生命週期**：父資料被刪/下架時，這些關聯資料怎麼辦（cascade？保留過濾？）？counter/冗餘欄位誰回減？（既有刪除路徑先讀 code 確認現況）
- 系統自動建立的資料（backfill、upsert 附帶建立），每個欄位填什麼？
- 資料量級（影響效能設計）？會不會無上限增長（要不要保留上限/清理）？

## 介面與整合
- 對外介面是什麼形式（REST/GraphQL/函式/事件/平台 channel）？
- 輸入怎麼驗證？驗證失敗回什麼？列表有分頁上下限嗎（沿用專案慣例的數字）？
- 相似操作之間語意一致嗎？刻意不同的地方是哪些、為什麼？
- 依賴哪些外部服務 / 其他團隊的 API？那些介面穩定嗎、有沒有 spec？
- 對外契約文件（給前端/他隊的 API doc）要不要跟著更新？誰維護、怎麼版本化？

## 行為與狀態
- 正常流程的步驟？
- 有沒有狀態機（草稿/送審/通過…）？合法的狀態轉換有哪些？非法轉換怎麼處理？
- 有沒有非同步 / 背景處理 / 排程？

## 邊界與例外（最容易被需求省略，務必逼問）
- 空值 / 空清單 / 超長輸入怎麼處理？
- 併發、重複提交、競態？
- 網路失敗 / 逾時 / 部分成功 → 重試？回滾？冪等性？
- 權限不足 / token 過期 / session 失效？
- 邊界值（0、負數、上限、分頁尾端）？

## 非功能
- 效能/延遲目標？
- 安全：敏感資料、加密、稽核 log？
- i18n / 多語 / 時區？無障礙？
- 需要哪些可觀測性（log/metric/告警）？

## 相容與風險
- 會不會破壞既有功能？向後相容要求？
- 已知風險或技術不確定性？**刻意不緩解的風險**有哪些（要顯式寫「接受＋理由」）？
- 上線方式：feature flag、灰度、一次全開？

## 交付與驗證（SDD）
- 交付物除了 code 還有什麼？（migration、契約文件、驗證腳本、輔助工具）
- 驗收怎麼跑？有本地測試環境嗎？沒有的話，驗證腳本要不要列為交付物？
