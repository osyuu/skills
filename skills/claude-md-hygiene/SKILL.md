# CLAUDE.md Hygiene — 常駐規範檔別腐爛

CLAUDE.md 每個 session 都被載入,是給 **agent** 的常駐規範,不是專案說明書。塞進易變內容 = 每次都重讀到過期、甚至**主動誤導**(規範叫 agent 去用一個已經被刪的 API)。這個 skill 的目標:讓規範檔只留「不隨實作變動的不變式 + 踩坑禁區」,易變的一律外移 + 指標。

## 何時用 / 何時不用

**用**:審查/重寫 CLAUDE.md、發現它混了狀態·待辦·契約·設計細節、替新專案訂常駐指南、診斷「這份規範怎麼一直過期」。
**不用**:把需求變設計書 → `design-doc`;寫應用程式 code → 一般開發;寫給人讀的 README/文件 → 不是這個。

## 三分類（決定每一行的去留）

1. **不變式規範 / 慣例 / 踩坑禁區 → 留 CLAUDE.md**
   - **不變式**:架構鐵律、命名、狀態範式、錯誤處理…「不論實作怎麼變都成立」的約束。
   - **踩坑禁區 / 決策理由**:「做 X 會爆 / 為何不選 Y / 這個 API 這版不可靠勿用」——穩定、防重犯,是規範檔**最有價值**的部分,別為了精瘦砍掉。
2. **易變內容 → 踢到活文件,CLAUDE.md 最多留一句指標**
   - 當期狀態 / 待辦 / 進度 → 專屬活文件(如 `docs/xxx-todo.md`)。
   - API 契約(欄位·端點) → 契約文件。
   - 完整設計 / 決策脈絡 → design docs。

外移時**只搬一處、不兩邊都留**——兩邊都有就是新的漂移源。

**指標要指得到(但別自作主張)**:CLAUDE.md 若會 committed,而指標目標是 git-excluded 或還沒進 repo 的檔(design doc / 個人筆記常被本機排除),對隊友 / CI 就是**懸空指標**。**發現就 flag 給使用者決定**(是刻意個人本機,還是某些該進版控)——**別擅自改指向、也別強制 commit**;這是專案相依、隨狀況變的決定。

## 判準（那把尺——每句寫進 CLAUDE.md 前先過）

一句話若寫死了以下任一,它就是**對當前 code 的斷言(狀態)**、會被下次改動證偽 → **不放** CLAUDE.md:

- **完成態**:已建 / 已退役 / 已移除 / 已上線 / 本期做到哪 / 尚未做。
- **名冊 · 數量**:「現有這幾個 repository / feature / 消費端」的清單。
- **route 清單 · 欄位清單**:精確列舉(會漂移,且常與契約檔重複)。
- **日期戳**:「20XX-XX-XX 改」。

**可以**提**穩定結構錨點**(「`app_router` 擁有路由」「共用 widget 在 `core/widgets/`」)——那是結構契約,不是「現在有哪些」的快照。

**休眠 / 已移除功能的特例**(最容易混):寫**啟用時的不變式**(現在式、當它是參考設計),把「當前開/關」放**指標**到活文件。**別在規範裡就地寫「X 目前休眠 / 本期不做 / 已移除」**——那是會翻轉的狀態。反例(不要):「登入目前不在本期範圍、seam 休眠」。正例:「gated 功能走單一 guard 入口(當前是否啟用見 `docs/xxx-todo.md`)」。

## 執行鐵律：重寫成不變式，不是逐行換皮

最常見的失敗:只逐行刪掉幾句明顯的狀態,結構沒動 → 看起來跟舊版沒兩樣、易變內容仍在。**這是換皮不換骨。**

若整份讀起來像「當前 code 的說明書」(現在式斷言哪個檔·provider·feature 存在) → **整份重寫成不變式約束**,別 nibble。
- 症狀:大量寫死路徑 / class 名 / 「已建·已退役」。
- 解法:每節問「這條的**不變式**是什麼?」——保留不變式,把具體現狀指到活文件。
- 檢查:重寫後跟舊版**結構應該明顯不同**(合併鬆散章節、砍名冊、指標化)。若「跟以前沒差別」,你只換了皮。

## 審查必做：grep-verify，別信散文

**規範檔的散文會過期而沒人發現。** 凡 CLAUDE.md 提到的 API / 檔 / route / provider,**逐一對 code 驗證還在不在**——別信文件講的。

真實踩雷:一份 CLAUDE.md 用現在式教 agent 呼叫**三個已被刪除**的東西(guard function、route、檔案),因為沒人 grep 驗過 → agent 照著 call 不存在的 API。

audit 起手式:
```bash
# 1. 撈完成態 / 日期戳
grep -nE '已建|已退役|已移除|已上線|本期|待接|尚未|休眠|20[0-9][0-9]-[0-9]' CLAUDE.md
# 2. 對每個被提到的 symbol / 檔案,驗證還在
grep -rl "<symbol>" <src>/ >/dev/null || echo "DEAD: <symbol> 已不存在，該從規範移除"
```

**交付前對你自己剛寫的草稿把上面兩個 grep 都跑一次**——別只驗 symbol 就交(#1 抓 dead ref、#2 抓漏網的完成態/狀態句,兩個都要,少跑一個就會有狀態句溜進去)。

**symbol 在 ≠ 主張還對。** grep 只證「symbol 存在」,證不了「踩坑禁區的技術主張還成立」。**你寫下或沿用的行為主張**(「靠 X 判完成」「用 Y 會爆勿用」)要**讀 code 確認現在還是不是這樣做**,不只驗存在。**繼承來的舊 landmine 最該驗**——最可能已語意過期(照抄舊規範沒驗 = 最常見踩雷:symbol 還在、主張卻反了)。成本分級:symbol 存在 grep 極便宜、一律驗;行為主張要讀 code、較貴,至少驗你保留/寫下的那些,不必回頭重推導全部。

連「某個 UI 接了沒」這種狀態都**極易判錯**(作者本人盯著 code 查都會連錯)——這正是狀態不該進規範檔的鐵證:狀態太易錯,放進每-session 載入的檔必爛。

## greenfield / 首份 CLAUDE.md（還沒 code、或 code 很少）

上面的 grep-verify(#2 驗 symbol 存在)**預設 code 已存在**。寫**第一份** CLAUDE.md 時方向相反——規範是意圖、code 之後才長,沒 symbol 可驗。這時:

- **需求還模糊 → 先收斂,別直接寫規範。** 用 **`design-doc`** 把模糊需求逼成決定,收斂後的**設計 / 已拍板決定就是真相來源**(取代「對 code grep-verify」)。design-doc 的「待確認 / 交付物」天然就是本 skill 要外移的易變內容集散地,CLAUDE.md 指過去即可。**本 skill 不自備收斂——辨識「需求還糊」→ 交棒 design-doc → 再回來寫規範**,別在這裡重造一個 lite 版收斂(兩處做收斂必然漂移)。
- **三條 greenfield 護欄**(grep 驗不到、但這是 greenfield 的真風險):
  1. **只鎖已拍板的**——founder 明講的 stack、想要的分層鐵律。
  2. **未定的標開放題 + 指標,別寫成事實**——後端選型 / 登入 / 離線這種「還沒決定」的,寫進規範就是假規範。用「休眠 / 開放題」那條:寫**啟用後的不變式**、把「選哪個」指到 design doc(例:「後端選型變動只准衝擊 repository 層以下」是不變式;「Firebase 還是自寫 REST」是指標)。
  3. **別幻覺技術選型**——founder 沒講的別腦補鎖死(只講 Riverpod,就別替他鎖 freezed / dio)。寧可規範薄,也別無中生有。
- grep-verify **#1(完成態 / 日期掃描)照跑**;**#2(symbol 存在)無 code 時 N/A**,改成「確認沒有任何 symbol 被當『已存在』寫死——命名一律用模板 / 佔位框架(`<noun>_repository.dart → NounRepository`),不是『現有 code 長這樣』」。

## 括號紀律（順手清）

句尾括號只在承載**理由 / 後果 / 你不知道會踩的約束**時才留：
- 留：「(否則 family 去重壞)」「(這版無 `valueOrNull`)」——講後果 / 原因。
- 砍：「(Riverpod)」——冗餘範圍標籤;「(統一,勿各寫各的)」——情緒強調、無新資訊。

## 黃金範例形態

一份好的常駐規範像:**路由分派表 + response 慣例 + DB 慣例 + 部署硬約束 + 踩坑**——每節數行、密而不腫,不鋪陳實作細節。寧可指標到活文件,也不在規範檔就地展開會變的東西。若手邊有同 monorepo 的另一份精瘦規範檔,拿它當範本對齊。

## 品質自檢

- 每一行過判準了嗎?有沒有完成態 / 名冊 / route / 欄位 / 日期漏網?
- 是**重寫成不變式**,還是只換皮(結構跟舊版一樣)?
- 提到的 API / 檔 / route 都 **grep 驗過**還在?有沒有 dead 引用?
- **沿用 / 寫下的 landmine 行為主張**有對 code 驗「現在還是不是這樣做」嗎(不只驗 symbol 在)?繼承來的舊主張特別要驗。
- 易變內容真的踢到活文件了嗎——**只一處、留一句指標**(而非兩邊都有)?
- 踩坑禁區 / 決策理由留住了嗎(別為精瘦砍掉有用的第三類)?
- 剩下的括號都承載理由 / 後果嗎?
- **若是 greenfield / 首份**:需求還糊有先 `design-doc` 收斂嗎?未定的(後端 / 登入…)都標開放題 + 指標、沒幻覺 stack、沒把 symbol 當「已存在」寫死?
